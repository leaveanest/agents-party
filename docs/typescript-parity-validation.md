# TypeScript Parity Validation

OSA-17 captured the evidence required before removing the Python implementation. The runtime is now the TypeScript Slack app; Python is not a fallback path.

## Automated Coverage

Run the full TypeScript gate before cutover:

```bash
vp check
vp run typecheck
vp test
vp pack
```

The current automated suite covers these parity boundaries:

- Health and server wiring: `tests/server.test.ts`
- Slack app registration, Events API deduplication, and retry metadata logging: `tests/slack/app.test.ts`, `tests/slack/events.test.ts`, `tests/slack/idempotency.test.ts`
- Slack replies, active thread routing, flag-reaction translation, generated-media upload/handoff, and AgentRunner success logging: `tests/slack/agentHandlers.test.ts`
- Slack message and attachment normalization: `tests/slack/messageHistory.test.ts`
- AI SDK message conversion for text, image, file, audio transcript degradation, tool calls, and unsupported attachments: `tests/providers/aiSdkMessageConverter.test.ts`
- Provider routing, capability checks, adapter selection, native provider lanes, and model registry coverage: `tests/providers/*.test.ts`
- AgentRunner tool rounds, structured result validation, and provider-backed generation: `tests/agents/*.test.ts`
- OAuth coordinators, encrypted context/token handling, OAuth HTTP routes, and one-time state consumption: `tests/integrations/oauth/*.test.ts`
- PostgreSQL migrations and repository behavior for Slack installations, routing, OAuth, workspace credentials, and RSS feed persistence: `tests/infrastructure/postgres/*.test.ts`

Set `TEST_DATABASE_URL` to run the gated PostgreSQL integration test against a real database. That test applies TypeScript migrations in an isolated schema and round-trips repository reads/writes.

## Observability

The TypeScript runtime emits structured logs for the highest-risk Slack and provider paths:

- Duplicate Slack Events API deliveries log `eventId`, `retryNum`, and `retryReason` before feature handlers run.
- Successful AgentRunner Slack executions log `eventType`, `teamId`, `channelId`, `threadTs`, `messageTs`, selected `provider`, selected `modelId`, structured-result presence, media kind, and tool result count.
- AgentRunner failures in app mentions, follow-up messages, and reaction translations log the error with Slack team/thread context.
- Thread routing persistence failures log team and thread context while still returning the Slack response.
- Generated media delivery logs whether Slack received a file upload, provider URI, or long-running operation handoff.

These logs are intentionally emitted at Slack handler boundaries so production support can correlate user-visible failures with provider selection, Slack retry behavior, and persistence failures without exposing message contents or secrets.

## Manual Slack Verification

Perform this checklist in a staging Slack workspace before major runtime or deployment changes:

- `GET /healthz` returns a healthy response from the TypeScript process.
- A fresh `app_mention` produces one threaded reply and logs the selected model/provider.
- Re-delivering the same Slack event id is suppressed and logs Slack retry metadata.
- A normal thread follow-up in an active routed thread auto-replies only when the channel and thread policies allow it.
- A follow-up in a disabled channel or with thread auto-reply disabled does not call AgentRunner.
- Adding `:flag-jp:` or another supported flag reaction to a text message posts a translation in the source thread.
- A web research request returns a sourced answer and records the web-search provider path.
- A Google Maps place, nearby, and route request each return typed map or route output.
- Image generation uploads a generated file to the Slack thread when provider bytes are returned.
- Video generation either uploads generated bytes or posts the provider URI/operation handoff when rendering is asynchronous.
- A text-only model with unsupported attachments fails clearly instead of silently dropping the attachment.
- Google OAuth and Salesforce OAuth start/callback flows reject replayed state and persist encrypted active connections.
- `vp run migrate` succeeds against a staging copy of production data and repository smoke tests can read/write the migrated tables.

## Rollback And Data Compatibility

Python removal was a destructive application cutover. After OSA-16:

- Rollback is restoring an older deployment artifact plus a database backup or compatible schema snapshot; the repository does not contain a Python fallback service.
- TypeScript migrations are additive/idempotent where possible, but production rollback after new writes must be validated against stored JSON payloads and index tables.
- OAuth state consumption is one-time delete-returning behavior. A rollback cannot safely replay consumed OAuth callback state.
- Generated media handoffs may include provider operation names or URIs that only the TypeScript provider path understands.
- Slack event deduplication is process-local unless a shared idempotency repository is introduced. Horizontally scaled rollback/roll-forward plans must account for duplicate delivery risk.
- Provider model ids and capabilities are registry-owned. Rollback to older code may fail if production configuration points to models added only in the TypeScript registry.
