Quick Start

まずはDocker Composeで、webとworkerまでまとめて確認する。

Agents partyのローカル確認は compose.yaml が最短です。Slack credentialsなしでも、PostgreSQL、Redis、migration、web、workerをまとめて起動し、 GET /healthz まで確認できます。

Required

ローカル前提

Node.js 22.x、vp、Docker Compose、Slackアプリを作成できるワークスペースを用意します。

Do not switch

パッケージ操作

コンテナ外で依存同期、検証、buildを行う場合は vp を使います。 通常操作で npm / pnpm / yarn へ切り替えません。

Secrets

秘密情報

Slack token、OAuth secret、provider API key、database URLを公開Issue、PR、ログ、スクリーンショットに載せないでください。

Composeで確認する

Slack連携なしのsmoke test、seedありのworkspace route、実Slack接続の順に進めます。

webとworkerを起動する

web と worker を指定すると、PostgreSQL、Redis、 migrationも依存関係として起動します。Slack credentialsが空でもhealth checkは確認できます。 

docker compose up --build web worker

health checkを確認する

別シェルでHTTPサーバーの疎通を確認します。ここまで通れば、アプリbuild、migration、 web processの最小経路は動いています。 

curl http://localhost:8000/healthz

最初のworkspace routeをseedする

app mentionをagent/modelへ解決するにはworkspace routeが必要です。ローカル確認では seed profileでbootstrapできます。実Slackで試す場合は AGENTS_PARTY_BOOTSTRAP_TEAM_ID を実workspaceのteam idにします。 

AGENTS_PARTY_BOOTSTRAP_TEAM_ID=T123456789 \
docker compose --profile seed run --rm seed

実Slack接続用に.envを用意する

Slack App Manifestの agents-party.example.com を http://localhost:8000 へ転送する公開HTTPSトンネルURLに置き換えてから、 Slack workspaceへimportします。Composeは .env を読みます。 

cp .env.example .env
# Fill SLACK_SIGNING_SECRET, SLACK_CLIENT_ID, SLACK_CLIENT_SECRET,
# SLACK_STATE_SECRET, AGENT_MODEL, and provider credentials as needed.
docker compose up --build web worker

ログと状態を確認する

Slack eventやprovider実行を確認するときはweb/workerのログを分けて見ます。 queue modeではwebがSlack requestを受け、workerがagent実行と返信を担当します。 

docker compose logs -f web
docker compose logs -f worker

停止する

環境を作り直したい場合だけvolumeも削除します。PostgreSQLのローカルデータも消える点に注意してください。 

docker compose down
docker compose down -v  # reset local PostgreSQL and Redis volumes

コンテナ外で個別に動かす場合

開発時にwebだけを手元で動かしたい場合は、vp install 後にPostgreSQL/Redisだけを Composeで起動し、vp run dev や vp run worker を使います。

PR前の検証

通常の変更では、format/lint、型チェック、テスト、packを通してからPRにします。

vp check
vp run typecheck
vp test
vp pack
