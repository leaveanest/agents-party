Quick Start

まずはSlackなしで起動し、次にワークスペースへ接続する。

Agents partyはSlackアプリ、PostgreSQL、モデルプロバイダー、必要に応じてRedisを組み合わせて動きます。 いきなり全部を接続せず、HTTP health check、DB migration、Slack OAuth、workspace routeの順に確認します。

Required

ローカル前提

Node.js 22.x、vp、Docker Compose、Slackアプリを作成できるワークスペースを用意します。

Do not switch

パッケージ操作

このリポジトリでは npm / pnpm / yarn を直接使わず、通常操作は vp で行います。

Secrets

秘密情報

Slack token、OAuth secret、provider API key、database URLを公開Issue、PR、ログ、スクリーンショットに載せないでください。

最短の確認順

Slack接続まで一気に進める前に、失敗箇所を切り分けられる順番で確認します。

依存関係を同期する

vp が見つからない場合は、別のパッケージマネージャーへ切り替えず、先に vp を直します。 

vp install

SlackなしでHTTPサーバーを起動する

Slack環境変数がなくてもサーバーは起動し、GET /healthz で最小の疎通を確認できます。 

vp run dev
curl http://localhost:8000/healthz

PostgreSQLを起動してmigrationを流す

Slack installation、OAuth state、workspace routing、暗号化されたworkspace credentialsはPostgreSQLに保存します。 

docker compose up -d postgres
export DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party
vp run migrate

Slack App Manifestをワークスペースに入れる

slack-app-manifest.yaml の agents-party.example.com を、 http://localhost:8000 へ転送する公開HTTPSトンネルURLに置き換えてからSlackへimportします。 

export SLACK_SIGNING_SECRET=...
export SLACK_CLIENT_ID=...
export SLACK_CLIENT_SECRET=...
export SLACK_STATE_SECRET=...
export DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party
vp run migrate
vp run dev

最初のworkspace routeをseedする

app mentionがagent/modelへ解決できるように、Slack workspaceのteam idを使ってbootstrap routeを作ります。 ローカルの初期modelは例として google:gemini-2.5-flash を使っています。 

DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party \
AGENT_MODEL=google:gemini-2.5-flash \
AGENTS_PARTY_BOOTSTRAP_TEAM_ID=T123456789 \
vp run seed:bootstrap

必要ならRedis workerを分ける

Slack ackとprovider実行を分離したい場合は、Redis queueを有効にしてweb processとworkerを別々に起動します。 

docker compose up -d redis
SLACK_AGENT_QUEUE_ENABLED=true REDIS_URL=redis://localhost:6379 vp run dev
REDIS_URL=redis://localhost:6379 \
DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party \
vp run worker

モデルとAPIキー

AGENT_MODEL はSlack routingがまだmodelを返さないときのbootstrap/fallbackです。 本番相当のmulti-workspace運用では、process-level provider keyではなく、暗号化したworkspace credentialsを使います。

PR前の検証

通常の変更では、format/lint、型チェック、テスト、packを通してからPRにします。

vp check
vp run typecheck
vp test
vp pack
