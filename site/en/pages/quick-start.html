Quick Start

Start without Slack, then connect a real workspace.

Agents party combines a Slack app, PostgreSQL, model providers, and optionally Redis. Bring those pieces online in order: HTTP health check, database migrations, Slack OAuth, then the first workspace route.

Required

Local prerequisites

Use Node.js 22.x, vp, Docker Compose, and a Slack workspace where you can create or install an app.

Do not switch

Package workflow

Use vp for normal JavaScript and TypeScript workflows. Do not switch to direct npm, pnpm, or yarn commands.

Secrets

Keep secrets out

Do not put Slack tokens, OAuth secrets, provider API keys, database URLs, or copied Slack transcripts in public issues, PRs, logs, or screenshots.

Recommended setup order

Check each layer separately so failures are easy to isolate before real Slack traffic is involved.

Install dependencies

If vp is missing, fix that first instead of changing package managers. 

vp install

Run the HTTP server without Slack

The server can start without Slack environment variables and exposes GET /healthz. 

vp run dev
curl http://localhost:8000/healthz

Run PostgreSQL and migrations

Slack installations, OAuth state, workspace routing, and encrypted workspace credentials are stored in PostgreSQL. 

docker compose up -d postgres
export DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party
vp run migrate

Configure the Slack App Manifest

Replace agents-party.example.com in slack-app-manifest.yaml with a public HTTPS tunnel that forwards to http://localhost:8000, then import it into Slack. 

export SLACK_SIGNING_SECRET=...
export SLACK_CLIENT_ID=...
export SLACK_CLIENT_SECRET=...
export SLACK_STATE_SECRET=...
export DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party
vp run migrate
vp run dev

Seed the first workspace route

App mentions need an enabled agent and model route. Use the Slack workspace team id for AGENTS_PARTY_BOOTSTRAP_TEAM_ID. The model below is only a local bootstrap example. 

DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party \
AGENT_MODEL=google:gemini-2.5-flash \
AGENTS_PARTY_BOOTSTRAP_TEAM_ID=T123456789 \
vp run seed:bootstrap

Split Redis-backed work when needed

Enable the Redis queue when Slack acknowledgements should be separated from provider execution. 

docker compose up -d redis
SLACK_AGENT_QUEUE_ENABLED=true REDIS_URL=redis://localhost:6379 vp run dev
REDIS_URL=redis://localhost:6379 \
DATABASE_URL=postgresql://agents_party:agents_party@localhost:5432/agents_party \
vp run worker

Models and API keys

AGENT_MODEL is the bootstrap or fallback model when Slack routing has not resolved a workspace, channel, or thread model. For production-like multi-workspace traffic, prefer encrypted workspace credentials over process-level provider keys.

Validation before a PR

Run the normal repository validation set before opening a pull request.

vp check
vp run typecheck
vp test
vp pack
